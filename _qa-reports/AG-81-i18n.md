1. **JSX text nodes** — `>([^<>{}\n]{2,})<` filtered to strings containing

4. **`console.{log,warn,error,info}('literal…')`**.
5. **Prop literals** — `label:`, `title:`, `placeholder:`, `message:`,

## 4. Step 4 — `scripts/i18n-extract.js`

Dry-run by default, never modifies source code, only writes under
`locales/`:

```
node scripts/i18n-extract.js            # dry-run, prints summary
node scripts/i18n-extract.js --write    # also merge placeholders into he.json
node scripts/i18n-extract.js --project payroll-autonomous
node scripts/i18n-extract.js --json     # JSON report only
```

Behaviour:
1. Walks each project under the ignore-list
   (`node_modules`, `dist`, `build`, `.next`, `.git`, `artifacts/generated`, …).
2. Applies the 5 heuristics from §2.
3. Emits `locales/_extracted.json` and `locales/_extracted.md`.
4. With `--write`, merges a `__extracted.<project>.<sha1>` entry per new
   string into `he.json` — **never** overwriting a value that already
   exists (no-delete policy).

Keys are SHA1-prefixed (8 chars) because auto-transliterating Hebrew to
Latin is lossy. A human reviewer later promotes them out of `__extracted`
into a real namespace like `payroll.save_slip`.

## 5. Step 5 — `scripts/i18n-validate.js`

Six checks, in order:

1. **Parse** — every `locales/*.json` must be valid JSON.
2. **Missing** — every key in `he.json` must exist in `en.json`, `ar.json`,
   `ru.json`. *Errors if violated.*
3. **Placeholder parity** — same `{name}` tokens in all four locales for
   the same key. *Errors if violated.*
4. **Bidi markers** — on `he.*` and `ar.*` values, Latin runs inside RTL
   text should be surrounded by RLM (`\u200F`) or LRM (`\u200E`).
   *Warning only — many keys use `{count}` which is a legitimate embed.*
5. **Orphans** — a key present in a locale but never referenced by a
   `t('key')` / `i18n.t('key')` / `$t('key')` / `useT('key')` / `translate('key')`
   call. *Warning.* With `--strict`, orphans become errors. Orphans are
   **never auto-deleted** — no-delete rule.
6. **Missing from locale** — any Hebrew literal in source code whose exact
   string isn't present as a value in `he.json`. *Warning.*

Usage:

```
node scripts/i18n-validate.js                  # full check, human output
node scripts/i18n-validate.js --strict         # orphans → errors
node scripts/i18n-validate.js --format=json    # machine-readable
```

Current run: **0 errors, 47 warnings**.

Breakdown of the 47 warnings:

| Type | Count | Reason |
|---|---:|---|
| `bidi` | 44 | Legitimate `{placeholder}` embeds. Treat as advisory. |
| `orphans` | 1 | Muted because no `t()` calls exist yet. |
| `missing-from-locale` | 1 | 810 hardcoded Hebrew strings — mostly comments; run `i18n-extract --write` to stub them. |
| `extra-in-locale` | 1 | Currently 0. |

**Gate status for CI: PASS.** Once `t()` calls start appearing, the
orphan detector un-mutes automatically.

## 6. Step 6 — Hebrew-specific concerns

### 6.1 Dates

- Existing `toLocaleString('he-IL')` calls found in:
  - `paradigm_engine/ui/nexus-ai-dashboard.jsx` line 47 — uses explicit
    `he-IL` locale. Good.
  - `paradigm_engine/paradigm-part4.js` lines 550, 606, 736 — good.
  - `techno-kol-ops/src/services/signatureService.ts:467` — good.
- `locales/he.json dates.*` provides canonical day/month names
  (`ראשון…שבת`, `ינואר…דצמבר`) plus `today/yesterday/tomorrow`.
- `__meta.firstDayOfWeek = 0` (Sunday) for he and en, `6` for ar (Saturday
  per some Arabic-Israel calendars — feel free to change to `0` if the
  product prefers the Israeli Sunday-start), `1` for ru.
- Recommended runtime helper in `docs/I18N.md` — wrap every date with
  `Intl.DateTimeFormat(locale, { dateStyle: 'short' })`.

### 6.2 Currency

- Symbol: `₪` (ILS / NIS). `__meta.currency.symbol` in every locale.
- Position:
  - he / en / ar — **prefix** (`₪ 1,234.56`) — matches existing code
    (`payroll-autonomous/src/App.jsx:41` uses `'₪ ' + …`).
  - ru — **suffix** (`1 234,56 ₪`) — matches Russian convention and
    ru-RU's default `Intl.NumberFormat` currency placement.
- Existing hardcoded `₪` occurrences (onyx-ai error messages) are captured
  in the `errors.*` namespace with placeholders; the translations keep the
  `₪` so the symbol appears in the expected spot per locale.
- Recommended runtime helper: `Intl.NumberFormat(locale, { style: 'currency', currency: 'ILS' })`.

### 6.3 Number formatting

| Locale | Pattern | Decimal | Thousands | Source of truth |
|---|---|---|---|---|
| he | `1,234.56` | `.` | `,` | matches existing `toLocaleString('he-IL')` |
| en | `1,234.56` | `.` | `,` | standard en-US |
| ar | `1.234,56` | `,` | `.` | standard ar-* (Israel uses Western digits) |
| ru | `1 234,56` | `,` | ` ` (NBSP) | standard ru-RU |

The Hebrew row is intentionally the same as en (Israel uses Western
grouping even though `Intl.NumberFormat('he')` may emit narrow-no-break
spaces — the app convention is comma grouping).

### 6.4 Address form ("לחץ" vs "לחצו")

Grep'd every `.{ts,tsx,js,jsx}` in the repo:

- **"לחץ" (second-person masc. singular — direct, neutral)** — 14 hits, all
  current production code (`MobileApp.tsx`, `VacationBalance.tsx`,
  `FinancialAutonomy.tsx`, `VoiceFab.tsx`, `gmail-service.ts`,
  `payroll-autonomous/src/App.jsx`, `bank-dashboard.jsx`, etc.).
- **"לחצו" (second-person plural)** — 0 hits.

**Policy decision (documented in `he.json → __meta.policy.addressForm`):**
use **"לחץ"** across every locale key. This is the existing house style
and matches the tone of the payroll slip, which addresses a single user.
The validator does *not* enforce this yet — a follow-up can add a lint rule
that rejects "לחצו" / "שמרו" / "בטלו" in PRs.

### 6.5 Spelling (כתיב חסר vs כתיב מלא)

Existing code is overwhelmingly **כתיב מלא** (full spelling, with yods):

- `חיסכון` not `חסכון`
- `סיכום` not `סכום`
- `עובדים`, `ייצור`, `חישוב`, `אישור` — all ktiv-male.

Declared in `__meta.policy.spelling = "ktiv-male"`.
`he.json` is internally consistent with this: every value uses ktiv-male.
Any future translator merging PRs must honour the policy.

### 6.6 RTL / bidi notes

- The `he` and `ar` locales declare `direction: "rtl"` in `__meta`.
- Mixed-direction runs (like `"הנפק PDF"` or `"חשבון {code} לא קיים"`)
  render correctly in modern browsers without manual RLM markers **as
  long as** the outer container is `dir="rtl"`. The extractor flags these
  as `[bidi]` warnings so a human can decide whether to inject explicit
  `\u200F` for email templates or PDF generators where the rendering
  engine is less forgiving.
- For the payroll PDF output (`onyx-procurement` route), we recommend
  rendering HTML with `<html dir="rtl">` and relying on CSS `unicode-bidi:
  plaintext` for mixed-content cells.

## 8. Key takeaways

1. The 4 projects are **i18n-greenfield** — zero runtime i18n exists, which
   means we can pick a canonical stack (i18next) without migrating away
   from anything.
2. The worst offender is `payroll-autonomous/src/App.jsx` — one file with
   every user-visible string baked in. Ironically this is also the cleanest
   to fix: one refactor replaces the whole app.
3. `onyx-ai` errors are a silent time-bomb: Hebrew in `throw new Error(…)`
   means English-locale tools (Sentry, DataDog, grep) see mojibake. Move
   them to error codes + `t('errors.*')`.
4. `GPS-Connect` is already i18n-clean (it's a generated API client) — no
   action needed.
5. Seed data in `AI-Task-Manager/lib/db/src/seed.ts` is **data**, not UI;
   it belongs in the DB and should not be treated as translation.
6. The Hebrew house style ("לחץ", ktiv-male) is declared in machine-readable
   form inside `he.json.__meta.policy` so any linter can enforce it.
7. Every generated file in this pass is **additive**. No source file was
   modified; no existing file was deleted.
