## 3. Book-to-Tax Adjustment Rules

Each rule in `ADJ_RULES` is a pure function `(input, context) → { amount, explanation }`.
A **positive** `amount` increases taxable income (an add-back). A **negative** `amount` reduces it (a relief).

| Rule              | Statutory basis                                               | Formula / Boundary                                                                             |
| ----------------- | ------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `entertainment`   | תקנות מס הכנסה (ניכוי הוצאות מסוימות), התשל"ב-1972 — כיבוד 80% | `nonDed = expense × (1 − 0.80)`; add back 20% of entertainment spend.                           |
| `donations`       | פקודת מס הכנסה §46                                             | Cap = min(30% × taxableBeforeAdj, 10,587,000₪). Donations below 200₪ → wholly non-deductible; donations above cap → excess added back. |
| `depreciationDiff`| תקנות פחת, התש"ט-1941                                          | `diff = bookDepreciation − taxDepreciation`. Positive = book > tax (add back); negative = deduct extra. |
| `fxSection9`      | פקודת מס הכנסה §9                                              | Unrealised FX on capital items is non-taxable. `amount = −fxUnrealisedCapital` — i.e. book gain is subtracted, book loss is added back. |
| `accrualVsCash`   | בחירת בסיס דיווח                                                | Caller computes delta; engine passes it through as the adjustment.                              |
| `vehicle`         | תקנות רכב                                                      | `nonDed = vehicleExpense × (1 − deductibleShare)`; default share = 45%.                         |
| `fines`           | פקודת מס הכנסה §17                                             | 100% of fines/penalties added back (never deductible).                                          |

### 3.1 Donation-cap boundary examples

| taxableBeforeAdj | donation | cap (30%) | excess added-back | test case                         |
| ---------------: | -------: | --------: | ----------------: | --------------------------------- |
| 100,000          | 50,000   | 30,000    | 20,000            | `above 30% cap adds excess back`  |
| 100,000          | 30,000   | 30,000    | 0                 | `exactly at 30% cap → zero excess`|
| 100,000          | 100      | 30,000    | 100               | `below floor rejected entirely`   |

### 3.2 Entertainment boundary examples

| entertainmentExpense | 80% deductible | 20% added back |
| -------------------: | -------------: | -------------: |
| 10,000               | 8,000          | 2,000          |
| 0                    | 0              | 0              |

## 4. Tax Computation

```
bookProfit          = revenues − cogs − opex − financialNet + extraordinaryNet
taxableIncome       = bookProfit + Σ adjustmentRule.amount
corporateTax        = max(0, taxableIncome) × 0.23
taxDue              = corporateTax − (advances + credits + withholdingCredits)
```

## 5. Output Formats

| Format | Renderer       | Consumer                                     |
| ------ | -------------- | -------------------------------------------- |
| JSON   | `renderJson()` | API response + DB persistence                |
| XML    | `renderXml()`  | Shidur envelope for רשות המסים submission    |
| PDF    | `renderPdf(result, outPath)` | Printable summary (`pdfkit` optional — falls back to a UTF-8 text file when pdfkit is not installed) |

The XML envelope includes:
- Header (formType, formVersion, year, preparedAt)
- Company (id, legal name, tax file number, fiscal year)
- Sections → nested Row elements with bilingual `LabelHe` / `LabelEn` + `Amount`
- IncomeStatement summary
- BalanceSheet summary
- Adjustments (rule name + amount + explanation)
- TaxComputation (book profit, taxable income, rate, corporate tax, advances, credits, withholding, tax due)

All text values are XML-escaped (`& < > " '`) — verified by the test `renderXml — escapes special characters in company name`.

## 7. Upgrade, Not Delete — Rule Compliance

- The existing reference `buildForm6111` in `src/tax/form-builders.js` is **untouched**. The new `src/tax/form-6111.js` extends the family with the calculation pipeline.
- `generate6111` clones input trial balance objects via value-only reads; a dedicated test (`input trial balance not mutated (never delete rule)`) snapshots the TB JSON before and after to prove no mutation.
- Contra accounts (acc depn, closing inventory, declared dividends, discounts returned) carry a `contra: true` flag so their credit-balance semantics are preserved instead of being overwritten.
- Every new constant lives in one place (`CONSTANTS_2026`) so future tax-year upgrades are additive, not destructive.

## 8. Known Limitations / Future Upgrades

1. **Schema-driven XML.** The current envelope is structurally valid and round-trips through any XML parser, but a live submission to שע"מ will require the exact XSD published by רשות המסים. Feed an XSD into a future agent to generate a validated serializer.
2. **Carry-forward losses.** `computeCorporateTax` records a loss number but does not persist it across years. A future `tax-carry-forward` module should tie into the company profile.
3. **Multi-entity consolidation.** This engine operates on a single company's TB. For group filings, wrap it in a consolidator agent.
4. **Deferred taxes.** Row 8300 is in the map but no automatic DTL/DTA computation yet — inputs must be provided manually via the TB.
5. **Corporate tax rate monitoring.** Rate has been 23% since 2018; if the Treasury changes it, update `CONSTANTS_2026.CORPORATE_TAX_RATE` and add a 2027 constants file — never patch 2026 values.
