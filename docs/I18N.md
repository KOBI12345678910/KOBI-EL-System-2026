# I18N — Internationalisation guide

**Owner:** Agent 81
**Last updated:** 2026-04-11
**Scope:** 4 projects — `payroll-autonomous`, `onyx-ai`, `AI-Task-Manager`, `GPS-Connect`.

This document is the house contract for internationalisation across the
monorepo. It is complementary to the Hebrew-first UX of the product:
Hebrew is the default, and every other locale is a shifted version of it.

---

## 1. Supported locales

| Code  | Name     | Direction | Default | First day | Notes |
|-------|----------|-----------|---------|-----------|-------|
| `he`  | עברית    | RTL       | **yes** | Sunday    | Israel's primary language. Source of truth. |
| `en`  | English  | LTR       | no      | Sunday    | Fallback for international users + dev tools. |
| `ar`  | العربية  | RTL       | no      | Saturday  | Arabic-Israel, Western digits, Gregorian. |
| `ru`  | Русский  | LTR       | no      | Monday    | Large Russian-speaking community in Israel. |

Currency is always **ILS (₪)**; only the number formatting and symbol
position change per locale (see §5).

---

## 2. File layout

```
locales/
  he.json            ← default. Full key set. 286 keys across 11 namespaces.
  en.json            ← identical key set, English values.
  ar.json            ← identical key set, Arabic-Israel values.
  ru.json            ← identical key set, Russian-Israel values.
  _extracted.json    ← auto-generated by `scripts/i18n-extract.js`.
  _extracted.md      ← human-readable version of the above.
scripts/
  i18n-extract.js    ← scans source, emits `_extracted.*`, opt-in merge into he.json.
  i18n-validate.js   ← CI gate. Run on every PR.
docs/
  I18N.md            ← this file.
```

Each locale file starts with a `__meta` block. Anything else under `__*`
is reserved — the validator ignores keys whose first segment starts with
a double underscore.

## 3. Key naming convention

```
<namespace>.<section>.<slug>
```

Examples:

```
common.save                → "שמור"
common.select_employee     → "בחר עובד…"
payroll.compute_preview    → "חשב תצוגה מקדימה"
payroll.hours_overtime_125 → "נוספות 125%"
errors.rfq_not_found       → "RFQ {id} לא נמצא"
confirmations.delete_item  → "למחוק את {name}?"
toasts.balances_moved      → "היתרות הועברו בהצלחה לשנת {year}"
dates.day_sunday_long      → "יום ראשון"
```

Rules:
- Slugs are `lower_snake_case`. No camel-case.
- Placeholders use `{name}` form, not `{{name}}` or `%s`. (The runtime
  wrapper translates to any format your i18n lib expects.)
- Keep namespaces flat — two levels max. If you need three, split into a
  new namespace instead of nesting deeper.
- Never reuse the same slug across namespaces unless the meaning is
  exactly identical.

## 4. Hebrew policy (declared in `he.json.__meta.policy`)

### 4.1 Address form

**Use "לחץ", not "לחצו".**

The product addresses a single user in second-person masculine-singular.
This is already the 100% convention in existing code — the policy only
codifies it. Other verbs follow the same pattern:

| English  | Canonical Hebrew |
|----------|-------------------|
| click    | לחץ               |
| save     | שמור              |
| cancel   | ביטול             |
| delete   | מחק               |
| confirm  | אשר               |
| back     | חזור              |
| continue | המשך              |

If you are translating from another language and are unsure, stay in
masculine-singular. Feminine forms and plurals are explicitly **not**
used in the UI at this time.

### 4.2 Spelling — כתיב מלא (ktiv-male)

Use full spelling with yods and vavs:

- חיסכון (not חסכון)
- סיכום (not סכום)
- איחסון (not אחסון)
- ייצור (not יצור)

### 4.3 Punctuation

- Use `"`straight double quotes`"` inside values. JSON quotes them as
  `\"`, which is fine.
- Use en-dash `—` (U+2014) for parenthetical phrases, not double
  hyphens `--`.
- Use `…` (U+2026) for ellipsis, not `...`.
- Percentages: stick the sign straight after the number — `125%`, not
  `125 %`.

### 4.4 ILS symbol (₪)

- Write it **after** an optional NBSP: `₪ 1,234.56`.
- Inside error messages with placeholders, keep it attached to the value
  it describes: `"חייב ₪{debit}"`.
- The ru locale moves it to the end per Russian convention:
  `"{amount} ₪"`.

## 5. Number, currency, and date formatting

### 5.1 Numbers

Use `Intl.NumberFormat` with the locale code from `__meta.locale`:

```js
import he from '../locales/he.json';
const nf = new Intl.NumberFormat(he.__meta.locale, {
  minimumFractionDigits: 2,
  maximumFractionDigits: 2,
});
nf.format(1234.5); // "1,234.50"
```

Patterns per locale:

| Locale | Example      |
|--------|--------------|
| he     | `1,234.56`   |
| en     | `1,234.56`   |
| ar     | `1.234,56`   |
| ru     | `1 234,56`   |

### 5.2 Currency

Always ILS. Symbol position comes from `__meta.currency.position`:

```js
import ru from '../locales/ru.json';
const fmt = new Intl.NumberFormat(ru.__meta.locale, {
  style: 'currency',
  currency: ru.__meta.currency.code, // "ILS"
});
fmt.format(1234.5); // "1 234,50 ₪"
```

### 5.3 Dates

```js
import he from '../locales/he.json';

const short = new Intl.DateTimeFormat(he.__meta.locale, { dateStyle: 'short' });
short.format(new Date()); // "11.4.2026"

const long = new Intl.DateTimeFormat(he.__meta.locale, {
  year: 'numeric', month: 'long', day: 'numeric',
});
long.format(new Date()); // "11 באפריל 2026"
```

Day and month names are also pre-translated under `dates.*` if you
prefer a static lookup.

## 6. Runtime integration — suggested approach

We recommend **i18next + react-i18next** for the React apps and a thin
Node wrapper for `onyx-ai`. None of the 4 projects currently ships with an
i18n runtime — this is opt-in, not required by the audit.

### 6.1 React / Vite (payroll-autonomous)

```bash
pnpm add i18next react-i18next
```

```jsx
// src/i18n.js
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import he from '../locales/he.json';
import en from '../locales/en.json';
import ar from '../locales/ar.json';
import ru from '../locales/ru.json';

i18n.use(initReactI18next).init({
  resources: { he: { translation: he }, en: { translation: en }, ar: { translation: ar }, ru: { translation: ru } },
  lng: 'he',
  fallbackLng: 'he',
  interpolation: { escapeValue: false },
  returnNull: false,
});

export default i18n;
```

```jsx
// Any component
import { useTranslation } from 'react-i18next';
function MyButton() {
  const { t } = useTranslation();
  return <button>{t('common.save')}</button>;
}
```

For RTL switching, set `document.dir` from `i18n.dir(lng)` whenever the
language changes:

```js
import i18n from './i18n';
i18n.on('languageChanged', (lng) => {
  document.documentElement.dir = i18n.dir(lng);
  document.documentElement.lang = lng;
});
```

### 6.2 Node backend (onyx-ai)

```bash
pnpm add i18next
```

```ts
// src/i18n.ts
import i18next from 'i18next';
import he from '../locales/he.json';
import en from '../locales/en.json';

i18next.init({
  resources: { he: { translation: he }, en: { translation: en } },
  lng: 'he',
  fallbackLng: 'he',
});
export const t = i18next.t.bind(i18next);
```

Then in every error site:

```ts
import { t } from './i18n';
if (!doc) throw new Error(t('errors.document_not_found', { id: documentId }));
```

### 6.3 Plain HTML / email templates (AI-Task-Manager/api-server)

Load the JSON directly and interpolate with a tiny helper:

```ts
import he from '../../locales/he.json';
function T(key: string, vars: Record<string, string | number> = {}): string {
  const flat = key.split('.').reduce<any>((a, k) => a && a[k], he);
  return String(flat ?? key).replace(/\{(\w+)\}/g, (_, k) => String(vars[k] ?? ''));
}
```

## 7. Adding a new key

1. **Add to `locales/he.json` first.** Hebrew is the source of truth.
2. Run `node scripts/i18n-validate.js`. It will fail with
   `[missing-in-en] …`, `[missing-in-ar] …`, `[missing-in-ru] …`.
3. Add the same key to the other three locales with a translation (use
   the English translation as an intermediate step if you don't read
   Arabic or Russian — the validator doesn't check translation quality,
   only completeness).
4. Placeholder parity is enforced: the three translations must contain
   the same `{name}` tokens as the Hebrew source.
5. Run `node scripts/i18n-validate.js` again. It should pass.

---

## 8. Adding a new locale

1. Copy `locales/he.json` to `locales/<code>.json`.
2. Update `__meta.locale`, `__meta.direction`, `__meta.displayName`,
   `__meta.numberFormat`, `__meta.currency.position`, `__meta.firstDayOfWeek`.
3. Translate every value. `__extracted.*` stays as-is (those are source
   strings waiting for promotion).
4. Add the code to `REQUIRED_LOCALES` in
   `scripts/i18n-validate.js` (one-line change).
5. Run both scripts — they should pass.

## 9. The no-delete rule

**This codebase does not delete keys.** Ever. If a key becomes orphaned
(`[orphan] payroll.xxx`), it stays. Only a human PR reviewer can decide
to remove it, and only after checking that no out-of-tree project
consumes it.

Mechanisms:

- `scripts/i18n-extract.js --write` only **adds** keys under
  `__extracted`. Never mutates existing values.
- `scripts/i18n-validate.js` raises warnings for orphans, not errors
  (unless `--strict` is passed, which some projects may opt into).
- The validator preserves `[extra-in-locale]` keys with a "keeping —
  no-delete policy" note.

## 10. CI integration

Minimal recipe — drop into the root pipeline:

```yaml
# .github/workflows/i18n.yml
name: i18n
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: node scripts/i18n-validate.js
      - run: node scripts/i18n-extract.js --json > extract.json
      - uses: actions/upload-artifact@v4
        with: { name: i18n-report, path: |
          locales/_extracted.json
          locales/_extracted.md
          extract.json
        }
```

Once the team is wiring `t()` calls in earnest, flip to `--strict`:

```yaml
- run: node scripts/i18n-validate.js --strict
```

## 11. Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `[placeholder-mismatch] en.foo.bar …` | You added `{count}` to the Hebrew but wrote "One request" in English. | Add `{count}` to all translations. |
| `[bidi] he.foo has embedded Latin run without RLM marker` | Legitimate if the Latin run is a placeholder or a product name ("PDF"). Warning only — ignore unless the string renders wrong in a specific renderer. |
| `[missing-in-ru] common.save` | You added a key to `he.json` and forgot to update `ru.json`. | Add it with a Russian translation. |
| "לחצו" sneaked into a PR | House style violation. | Rewrite in second-person-masculine-singular: "לחץ". |
| A new locale flag reads the wrong format | `__meta.numberFormat` / `__meta.currency.position` mismatch. | Update `__meta`, then restart the dev server — `Intl.NumberFormat` caches at module load. |

## 12. Out of scope

The following are intentionally **not** part of this audit:

- **Plural rules** beyond ICU `{count}` placeholders. If the product ever
  needs "1 request" vs "2 requests" split, add `foo_one` / `foo_many`
  keys using i18next's built-in pluralisation — the validator will not
  complain.
- **Translation memory** / `.po` export. When needed, run
  `i18n-extract.js --json` and pipe into whatever TMS the team picks.
- **Machine translation fill-in.** The translator is a human. The
  scripts exist to make missing/orphan keys impossible to ignore, not
  to write translations for you.
- **Runtime language detection from `Accept-Language`.** Add that in
  your server / client bootstrap; it is not part of the locale files.
